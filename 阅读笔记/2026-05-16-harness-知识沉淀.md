## 一、文章主旨

2026 年初 Harness Engineering 热潮里，业界争论"模型多大""工作流多复杂"，作者团队反其道而行——

> **工作流是管道，知识才是流过管道的活水。**
> Skill / Agent / 工具链会随模型迭代，但**领域知识是永恒的**。

模型会换、工具链会更新、工作流会重构，但团队在**特定业务领域**积累的领域模型、架构决策、踩坑经验——这才是不会因模型换代而失效的护城河。

## 二、可移植的方法论三件套

### 1. 知识三维正交

| 维度 | 取值 |
|---|---|
| **存储层** | Layer 0-P 个人 / 0-T 团队约定 / 1 通用技术 / 2 业务领域 / 3 项目专属 |
| **类型（MECE）** | `model` 实体 · `decision` 决策 · `guideline` 推荐/禁止 · `pitfall` 坑 · `process` 流程 |
| **成熟度** | `draft` → `verified`（1 人/1 项目验证）→ `proven`（≥2 人/≥2 项目） |

> Layer 3 项目知识可**向上提升**——被判定为跨项目通用，自动升到 Layer 1/2。

### 2. 独立 Git 知识仓库（不寄生业务项目）

```
team-knowledge.git
├── knowledge-catalog.md          ← Agent 查询入口
├── .knowledge-config.yaml         ← 团队配置（成员、冲突策略）
├── team-conventions/              ← Layer 0-T
├── tech-wiki/                     ← Layer 1
│   ├── catalog.md
│   ├── patterns/TK-PAT-001.md
│   └── anti-patterns/
├── biz-wiki/                      ← Layer 2
│   └── {domain}/
└── contributions/
    ├── pending/
    └── conflicts/
```

**为什么独立 repo**：跨项目共享 / 生命周期独立 / 权限独立。

**三种角色**：maintainer（裁决）/ contributor（自动贡献）/ reader（只读）。

**区块链思想 + Git 实现**：log.md 只追加；`evidence.contributors[]` 溯源；共识机制驱动成熟度提升。

### 3. 三级渐进式索引（解决 Agent 上下文膨胀）

| 层级 | 文件 | 大小 | 作用 |
|---|---|---|---|
| Layer A 全景目录 | `knowledge-catalog.md` | ~50 行 | "知识库有什么？" |
| Layer B 分类清单 | 各目录 `catalog.md` | ~100-300 行 | "这类有哪些条目？" |
| Layer C 完整条目 | `TK-*.md` / `BK-*.md` | ~50-200 行 | "这条说了什么？" |

> Agent **不被推送**知识，而是**主动按需查**；各阶段有独立查询预算。
> 对比"一次推 50 条完整知识"（5000-10000 行），上下文效率提升一个数量级。

## 三、工作流与知识的耦合

```
INIT (git pull 知识 + 注入查询入口)
  ↓
各阶段执行（Agent 按预算查知识 + 在产物里记录 knowledgeReferences）
  ↓
ARCHIVE（@archiver 自动从产物提取知识 + 提升判定 + git push）
```

每次需求交付，本质上都是一次**知识沉淀**。下个人的 `/flow-run` 自动站在前人肩上。

---

## 四、关键工程哲学

- **"文件系统即状态机"**：所有产物 / 状态 / 知识都是 markdown 文件，无数据库无平台依赖，IDE 原生识别（`.codebuddy/` 目录）。
- **自动衰减机制**（借鉴 Karpathy LLM Wiki）：
  - proven 12 月未引用 → verified
  - verified 6 月未引用 → draft
  - draft 长期未用 → 归档
- **Lint 机制**：每 10 个工作流 / 30 天 / 手动 触发；检索矛盾、孤儿、过时、重复条目。

## 五、第八章："突破人机交互瓶颈"

腾讯内部 **Hapi 内网版**——手机 / PWA 在外网或通勤路上接管开发机 Agent 会话。

**脱离 Hapi 仍成立的工程启示**：
- 状态持久化（文件系统而非内存）
- 断点恢复
- 异步审批（提交→等待→任意设备审批）
- 通知触达（企微 / 邮件主动 push 而非被动检查）

## 六、我的判断 ⭐

### 强的地方

1. 抓到了 Harness 讨论里被忽略的"**知识维度**"，与"工作流编排"/"模型大小"形成第三极
2. 自动衰减 + Lint 借鉴 Karpathy，工程上有理论参照
3. 三级索引解决了 Agent 上下文膨胀的真实痛点

### 弱 / 理想化的地方

1. **知识自动提取的质量是整个体系的命门**，文章却避而不谈——`@archiver` 凭什么能产出高质量的 decision/pitfall/guideline？
2. "维度正交"在落地时**边界很模糊**——一条 decision 经常同时是 pitfall + guideline，五类型 MECE 是 PPT 友好的说法
3. **Layer 3 → Layer 1/2 的提升判定逻辑没展开**：谁来判"是否项目特有"？怎么判？人审还是 Agent 自动？

### 腾讯味重

- TAPD / iwiki / 企微 / Hapi / Codebuddy / TencentDB / 广告业务案例
- 外部团队要复用需要把基础设施（知识库 git + IDE 集成 + 通知通道）整套搭起来，工程门槛不低
- 第八章的"Hapi 内网版"是产品推广，但思路本身有迁移价值

### 最有移植价值的三件套

1. **知识独立 git 仓库**（不寄生业务项目）
2. **三级渐进式索引**（catalog → classification → entry）
3. **自动衰减 + Lint**（防止知识库腐烂）

## 七、对画画布 / 影视创作工具箱的启发

> 单人 / 小团队场景的最小可行版本：

1. **不需要 Layer 0-P / 0-T 区分**——一个人就是团队，统一一份约定
2. **只保留三层**：通用技术 / 业务领域 / 项目专属
3. **手动写知识，自动衰减**——不要追求 Agent 自动提取（质量未经验证）
4. **核心是三级索引**：写一个 `knowledge-catalog.md` 入口，Agent 启动时只读这一份
5. **git 仓库可以是项目内的 `docs/knowledge/`**——等到第二个项目需要复用时再独立

> Hapi 那段对个人开发者意义不大，但"状态持久化在文件而非内存"的哲学值得抄——所有 plan / progress / 决策都写 markdown，不依赖任何 GUI 状态。
