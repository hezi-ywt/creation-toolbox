# story-creator Skill 设计文档

> 日期：2026-05-17
> 状态：设计稿，准备实现
> 设计模式：Pipeline + Inversion + Tool Wrapper（参考 Google ADK 五种模式）

## 1. 目标

一个**创作者主权**的故事生成 skill。让创作者从模糊火花（一句话/一个画面/一个角色/一首歌的感觉）发展出可评估的故事胚胎（logline → premise → synopsis），用于创意起步阶段，**不是**最终影视剧本生产工具。

## 2. 核心理念

跟 AI 协作的好工具不是「AI 写得多好」，而是「让创作者听见自己脑子里没说出来的东西」。

AI 扮演三个角色，**作者始终是作者**：

- **挖矿工（Excavator）**：问"为什么是这个而不是那个"的好问题，把创作者脑子里没意识到的东西挖出来
- **分岔者（Diverger）**：从创作者的选择出发，给真正不同的方向（不是同质化候选）
- **挑刺者（Critic）**：给方向找坑——套路、动机洞、情绪曲线断、爽点稀疏

三角色**软切换**，AI 自然对话，不打 [Excavator]/[Diverger]/[Critic] 标签。

## 3. 状态模型

**两态**：

- **草稿**（默认）：随时改、丢、重写
- **已定**：创作者显式说"定了"才会进入

**不做**：备选分支、三层分层、依赖图、对话记录、metadata、版本号。

## 4. 文件结构

**用户视角**：

```
故事/
└── <故事名>.md
```

每个文件：

```markdown
# <故事名>

## 已定
（持久的事实——世界规则、主线、关键角色。改这里 AI 提醒影响范围。）

## 草稿
（在弄的——所有讨论写在这里。随便改、丢、重写。）
```

**没了**。一个文件 = 一个故事的全部状态。

## 5. AI 三个行为

### 行为 ①：正常讨论

- 默认在「草稿」段写
- 三角色软切换，按需要问 / 分岔 / 挑刺
- 不主动建议"该锁了吧"——错锁的代价远大于多次提醒
- 调用 `references/excavation-questions.md` 拉好问题
- 调用 `references/genre-conventions.md` 了解类型惯例
- 调用 `references/anti-cliches.md` 标记潜在套路

### 行为 ②：听到"定了"的意思

**自然语言识别**（不要求关键词）：

| 创作者可能这么说 | 识别为 |
|---|---|
| "这个就这么定了" / "定了" / "以后别改这条" / "这是基本设定" / "锁住" | 想提升为「已定」 |
| "我又改主意了" / "其实可能不是这样" / "回退" | 想从「已定」拉回「草稿」 |

**AI 的工作**：复述确认意图 + 问范围（这是唯一的"填表"）：

```
你："这个就这么定了"
AI："要把『主角是机器人』放进已定吗？这是 ——
  - 世界设定（世界里存在机器人，往后所有故事受影响）
  - 故事设定（仅这部作品里他是机器人）
  - 场景设定（仅这一场戏的反转）
  你选一个？"
```

范围是**唯一**显式信息，因为这从对话猜不出来。其他全靠 AI 推断。

### 行为 ③：要改「已定」时

**触发**：创作者修改的内容跟当前「已定」段冲突，或显式说"改 X"。

**流程**：

1. AI 用 grep 在当前故事文件全文搜索这条已定事实的关键词
2. 列出所有可能受影响的位置（在已定 / 草稿 / 其他段）
3. 把这些位置的现状贴给创作者，**逐条**让创作者决定 留 / 改 / 再聊
4. **不阻塞**——如果创作者说"先这么改，依赖回头再说"，AI 直接改并把"待审依赖"挂在文件顶端

```
你："其实他妈妈不是死了是失踪"
AI："这条在已定。文件里 2 处提到她死亡：
  - 草稿第 3 段：『他每年清明给妈妈烧纸』
  - 已定/主角动机：『失去母亲让他走上复仇之路』
  这两处贴给你，逐条说留 / 改 / 再聊。或者你说『先这么改，回头再说』我直接改。"
```

## 6. 触发条件（description 写法）

**严格遵循 Obra/writing-skills 的规则：description 只描述触发条件，绝不总结工作流。**

✅ 好例子：
> Use when the user wants to brainstorm or develop a story from a seed idea — a phrase, a scene, a character, a mood. Use for creative ideation phase, not for writing final scripts or screenplays. Trigger keywords: 故事点子, 想个故事, 想写故事, 剧本灵感, premise, logline, synopsis, brainstorm a story.

❌ 不要：
> Use when ... — uses three roles (Excavator/Diverger/Critic) to interview user, generate options, critique clichés...

## 7. references/ 内容

每份 100-300 行，AI 按需加载。

### excavation-questions.md
- 按种子类型分类的好问题（一句话 / 一个场景 / 一个角色 / 一首歌的感觉）
- 每条问题带"为什么这样问"的简短说明（避免 AI 死板套用）

### beat-structures.md
- Save the Cat 15 节拍
- 三幕剧
- Pixar Pitch 6 句模板
- 英雄之旅简化版
- 短剧节奏（中国市场，钩子+反转+爽点密度）
- 每种结构带"何时用 / 何时不用"的简短判断

### genre-conventions.md
- 主流类型：悬疑 / 爱情 / 治愈 / 喜剧 / 古装 / 科幻 / 武侠 / 恐怖 / 现实主义
- 每种类型：核心爽点、读者期待、常见禁忌、推荐节拍结构
- 中国市场偏好特别标注

### anti-cliches.md
- AI 高发套路（按频率排序）：失忆、灵魂互换、龙傲天、穿越、虐恋、扮猪吃老虎、玛丽苏...
- 每条：典型表现 + 为什么烂 + 免坑思路 + 优秀反例
- 重要：**不是禁用清单**，是"套路本身不死，烂用才死"——AI 标记后让创作者选

## 8. 测试策略

### 触发测试
20 个 prompt，10 个应该触发，10 个不应该：

应触发：
- "我有一个故事想法" / "想写一个雨夜便利店的故事" / "帮我想个故事" / "一个失忆的杀手"...

不应触发：
- "帮我把这个故事大纲写成剧本格式"（已超出 ideation 阶段）
- "写一个 Python 函数"（无关）
- "解释下三幕剧"（教学需求，非创作需求）

### 行为测试

| 场景 | 期望行为 |
|---|---|
| 创作者输入"雨夜便利店的场景" | Excavator 上场，提问挖深 |
| 创作者继续描述孤独老人等死去儿子 | Diverger 给 3 个真正不同方向 |
| 创作者选了"治愈/器官受赠者认亲" | Critic 标记这是套路 + 给免坑思路 |
| 创作者说"这个设定就这么定了" | 复述确认 + 问范围（世界/故事/场景） |
| 创作者后来说"其实改成失忆" | grep 全文 + 列影响 + 不阻塞 |

### 边界测试
- 创作者只给 5 个字："我想写故事" → AI 不应该编一个，应该追问
- 创作者给完整 logline → AI 应该跳过 Excavator，直接 Diverger 给premise方向
- 创作者改一条不在已定的事实 → AI 不应该触发依赖检查，正常 ack

## 9. 不做的事（明确边界）

- ❌ 不写最终剧本格式（场景头、对白格式、镜号）—— 那是后续 skill 的事
- ❌ 不做分镜 / shot list
- ❌ 不做多个故事的索引或交叉引用
- ❌ 不强制 AI 角色显式标签（软切换）
- ❌ 不维护结构化元数据（grep 够用）
- ❌ 不自动锁定（错锁代价高）
- ❌ 不阻塞修改（不打断心流）

## 10. 衡量是否做得好

skill 实现后，问自己三个问题：

1. **创作者主权**：所有关键创作决定是创作者做的还是 AI 替决定？应该是前者。
2. **意外发现**：使用过程中创作者是否说出了自己没意识到的偏好/想法？应该是。
3. **零词典**：创作者是否能用日常说话方式跟 skill 交互？应该是。
