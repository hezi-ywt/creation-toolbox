# 交付物格式规范

> 日期：2026-05-17
> 状态：v0.1.0
> 适用：影视创作工具箱项目

## 核心原则

1. **给程序的用英文，给人的用中文**——`.agents/`、`docs/`、`scripts/` 等技术目录英文；`阅读笔记/`、`故事/`、`参考资料/` 等用户内容目录中文。
2. **一件事一个文件**——别合并不同主题，也别为同一件事造两份不同口径的文档。
3. **活文档不带日期，归档/快照带日期**——SKILL/references 是活的，阅读笔记/设计文档/交付报告是当时的快照。
4. **AI 读的强制 frontmatter，人读的用一段 blockquote header**——YAML 是 AI 协议，人不要被它打扰。
5. **凡是规则要"以防万一"才存在的，就不存在**——这份规范本身也按这条裁。

## 目录结构

```
影视创作工具箱/
├── .claude/                      # Claude Code 项目级目录（自动扫描）
│   └── skills/<skill-name>/      # 项目自建 skill 放这里
├── .agents/                      # skills-cli 第三方 skill 装在这里
│   └── skills/<skill-name>/      # 比如 web-access（第三方）
├── docs/                         # 工程文档（英文目录，文件可中可英）
│   ├── 交付物格式规范.md         # 元规范（本文件）
│   ├── specs/                    # 设计文档（决策来龙去脉）
│   │   └── YYYY-MM-DD-<topic>.md
│   └── deliveries/               # 交付报告（一次性，可归档可删）
│       └── YYYY-MM-DD-<skill>.md
├── scripts/                      # 自动化脚本（按需出现）
├── 阅读笔记/                     # 外部文章/书籍读后笔记
│   └── YYYY-MM-DD-<主题>.md
├── 故事/                         # story-creator 输出
│   └── <故事名>.md
├── 参考资料/                     # 杂项参考（剧本片段、素材链接等）
└── skills-lock.json
```

**什么时候新增顶层文件夹**：只有当至少 3 份内容不属于现有任一文件夹时再加。不预先建空文件夹。

## 文件类型清单

### 1. SKILL.md（Anthropic 规范，固定）

```yaml
---
name: <skill-name>
description: <只写触发条件，不总结工作流>
license: MIT
metadata:
  author: 影视创作工具箱
  version: "x.y.z"
---
```

正文用中文。长度目标 < 300 行，超过就把长内容拆到 `references/`。

### 2. references/*.md（仅一行 frontmatter）

```yaml
---
description: <一句话说这份文件是干什么的——AI 决定要不要读>
---
```

每份 100-300 行。

### 3. 设计文档（docs/specs/YYYY-MM-DD-*.md）

人读的，开头一段 blockquote：

```markdown
# <标题>

> 日期：YYYY-MM-DD
> 状态：设计稿 / 已实现 / 已废弃

## <章节按内容自由组织>
```

长度目标 100-300 行。讲**决策和理由**——为什么这么做、不这么做的原因。**不**写"怎么用"（那是 SKILL.md 或交付报告的事）。

### 4. 交付报告（docs/deliveries/YYYY-MM-DD-*.md）

```markdown
# 交付报告：<skill 或功能名>

> 日期：YYYY-MM-DD
> 状态：vX.Y.Z

## TL;DR
（≤ 3 句）

## 怎么用
（起手命令 + 一个例子）

## 文件清单
（路径树）

## 我做的判断
（自主决定的事，留给用户 review）
```

长度目标 < 150 行。**不**重复设计文档的"为什么"——如果想说为什么，link 到设计文档。

之前的 `完成报告-story-creator.md`（项目根）按此规范要挪到 `docs/deliveries/2026-05-17-story-creator.md`。

### 5. 阅读笔记（阅读笔记/YYYY-MM-DD-<主题>.md）

```markdown
# <自拟标题，不抄原文>

> 作者：xxx
> 来源：xxx
> 链接：xxx
> 阅读日期：YYYY-MM-DD

## 一、文章主旨
## 二、可移植的方法论
## ...（按文章自由组织）

## 我的判断 ⭐
（强制章节——这是笔记和"摘抄"的分界线）

## 对本项目的启发
（强制章节——读了但没用上等于没读）
```

长度目标 100-200 行。

### 6. 故事文件（故事/<名>.md）

由 story-creator 维护，结构见 SKILL.md，不重复定义。

### 7. memory 文件（~/.claude/.../memory/*.md）

AI 协议规定的格式，遵循 Anthropic memory 工具规范：

```yaml
---
name: <短名>
description: <一句话>
type: reference | preference | project | ...
originSessionId: <自动填>
---
```

每份 30-80 行。MEMORY.md 索引每条一行。

## 命名规则

- **路径**：技术目录英文，用户目录中文。
- **文件名**：跟所在目录走——`docs/specs/` 下英文小写连字符；`阅读笔记/` 下中文。
- **时间戳**：`YYYY-MM-DD-` 前缀只用于"当时的快照"类（specs / deliveries / 阅读笔记）。活文档（SKILL.md、references、memory、故事）不带日期。
- **大小写**：英文文件名一律小写连字符（kebab-case），不用驼峰、不用下划线。

**Trade-off 自陈**：中文路径在某些跨平台 shell/CI 场景下需要 UTF-8 编码声明，少数老脚本可能炸。本项目是本地单人 Windows + Git Bash 环境，验证过 git/PowerShell/Bash 都正常处理，**接受这个代价**换"路径即文档"的可读性。如果某天要上 CI 或开源，再考虑迁英文。

## 引用 / Link 规则

- **能链就链一次，不要重复信息**：交付报告 link 到设计文档而不是复述。
- **不要强制互链**：阅读笔记之间没必要互相 link，除非真的在讨论同一件事。
- **link 格式**：相对路径（`../specs/2026-05-17-story-creator-design.md`），不写绝对路径。
- **memory 索引**（MEMORY.md）每条一行，格式 `- [标题](文件名) — 一句话摘要`。

## 不做的事

- ❌ 不建 `knowledge-catalog.md` / 三级索引（Harness 风格）——单人项目不需要
- ❌ 不引入成熟度标签（draft/verified/proven）——一个人写没人能 verify
- ❌ 不自动衰减归档机制——文件留着不碍事
- ❌ 不强制每份文档都有 changelog——git 就是 changelog
- ❌ 不为"以后可能有别的项目复用"提前抽象——等真有第二个项目再说
