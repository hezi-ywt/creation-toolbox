# 资产防污染参考

故事项目化后，最大的工程问题是**剧本内的资产一致性**：

- 角色 A 在不同场景里被叫成不同名字（"男主" / "李雷" / "他"——AI 当成三个人）
- 改了角色名，引用它的场景大纲 / 对白没同步
- 角色设定互相矛盾（已定段写 30 岁，草稿写 50 岁）
- 场景大纲引用的角色文件根本不存在
- 不同故事共用元素但设定漂移

**5 个污染场景，4 条防御规则**。

## 4 条 hygiene 规则

### 规则 1：项目隔离

**一故事一目录**。跨故事不共享资产（即使重名）。

```
故事/找回/角色/老王.md       # 故事 A 的老王（警察）
故事/江湖路/角色/老王.md      # 故事 B 的老王（车夫）—— 独立
```

**理由**：避免 AI 串戏，避免改一处影响多处。

**例外**：明确要复用（如系列剧），用 `世界观/` 共享 + 在每个故事 `index.md` 顶部声明引用源。

---

### 规则 2：唯一性约束（同故事内）

同一故事内同一类型资产 `title` 必须唯一。

如果两个角色都叫"老王"（编剧故意），加 **disambiguation suffix**：

```
角色/老王-警察.md   →  title: 老王（警察）
角色/老王-车夫.md   →  title: 老王（车夫）
```

文件名带 suffix，frontmatter `title` 带括号注释。

**为啥不直接改名**：因为编剧创作时"老王"已经在脑里固定了，强行改成"老李"反而失去人物感。Disambiguation 保留原名 + 加上下文。

**检测**：装了 `asset-index-cli` 后 `asset-index list --type 角色` 列出来，肉眼 + AI 双重看是否重名。

---

### 规则 3：改名联动（critical）

改资产名字时，必须**全局 grep + 提示用户更新引用**。这是 SKILL.md 行为 ③ 的扩展。

#### 触发
- 用户说"男主改名叫林默"
- 用户说"把停车场改成天桥"
- 用户说"把'死党'这个 disambiguation 去掉"

#### 流程

```
1. AI 用 grep 在故事目录里找所有引用：
   - 角色/*.md 里的关系段
   - 场景大纲/*.md 里的出场角色 + 场次描述
   - 草稿/*.md 里的讨论
   - index.md 里的人物表

2. AI 列出所有命中位置 + 当前上下文：
   "改 '男主' → '林默' 会影响：
   - 角色/男主.md: title 字段 + name 字段 + 文件名
   - 角色/女主.md: 关系段提及 '丈夫男主'
   - 场景大纲/主线.md: 5 个场次的角色字段 + 14 处场次描述
   - 草稿/讨论-2026-05-15.md: 8 处历史讨论"

3. 让用户**逐类决定**：
   - 角色 / 场景大纲 → 全改（正式资产，必须一致）
   - 草稿 → 留（历史痕迹，保留"叫男主时期"的讨论）
   或：用户说「先这么改，回头再说」 → AI 直接全改 + 标"待审"挂顶端

4. 改完后建议跑：
   asset-index check         # 验证一致性
   asset-index search 男主    # 确认 grep 不再命中老名字
```

#### 关键决策（让用户拍板）

- **是改 title 还是改文件名**？两者都改是默认。只改 title 保留文件名旧值适合"显示名换了但内部 ID 不变"
- **历史草稿改不改**？默认不改（保留思路演化痕迹）

---

### 规则 4：状态一致性 gate

资产 `status: 已定` 后，改它必须走完整的"行为 ③ 改已定"流程（详见 SKILL.md）：

1. grep 找所有引用
2. 列出受影响位置
3. 让用户逐条决定
4. 用户可说"先这么改，回头再说"——AI 直接改 + 加待审清单到 index.md 顶部

```markdown
# 找回

> ⚠️ 待审依赖（2026-05-17）：
> - 男主名字从 '男主' 改为 '林默'（已批量改 角色 / 场景大纲，草稿未改）
> - 场景大纲 E01 第 3 场改为侧面描写（旧版仍在 草稿/讨论-2026-05-15.md）

## 已定
...
```

**不阻塞**——用户优先，AI 标记后续审。

## 5 个污染场景对照表

| 场景 | 规则 | 怎么解 |
|---|---|---|
| 1. 角色在不同文件里叫不同名 | 规则 2（唯一性）| AI 每次提到角色用 `title` 字段；grep 查歧义 |
| 2. 改角色名引用不同步 | 规则 3（改名联动）| 改名前先 grep + 列影响范围 |
| 3. 设定互相矛盾 | 规则 4（状态一致）| 改 `已定` 走完整流程；规则冲突挂"待审" |
| 4. 场景引用不存在的角色 | 规则 2 + 改名联动 | `asset-index check` 检查 + AI 主动建文件或提示 |
| 5. 跨故事元素漂移 | 规则 1（项目隔离）| 不要跨故事共享；要复用就显式 import |

## 跟 asset-index CLI 联动

装了 CLI 后，`hygiene` 的检查可以半自动化：

```bash
# 找重名角色
asset-index list --type 角色 | sort | uniq -d

# 找出所有"已定"资产
asset-index list --status 已定

# 验证 frontmatter 一致性
asset-index check

# 找出引用某角色的所有文件（asset-index search 关键词）
asset-index search "林默"
```

**没装 CLI 时**：AI 用原生 grep + Read 做同样的事，只是慢一点。

## 反面教材

❌ AI 自作主张改资产名（不通知用户）—— 违反创作者主权
❌ 改了名字不 grep 不通知 —— 留一堆 dead reference
❌ 同故事内 2 个"老王"不加 disambiguation —— AI 必串戏
❌ 跨故事抄一份"老王.md" —— 设定一漂移就垮
❌ 用 `status: 半成品` 这种自创枚举 —— 见 [asset-rules.md](./asset-rules.md) 标准枚举
